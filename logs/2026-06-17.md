### 4. Click-Anywhere Navigation

Replaced the SVG prev/next button pair with a "click anywhere to continue" system.

**How it works:**

- The `tut-nav` div holds a single `tut-tap-hint` element (previously a dot, now the text "click to continue").
- A single click listener on the full-screen overlay handles all navigation:
  - If `introPlaying` is true → click ignored
  - If on the last page → `window.__startGame()` (or `window.location.href = 'index.html'` in standalone)
  - If `e.clientX < window.innerWidth * 0.25` → go back one step (unless `currentStep === 0` or `noPrev` is set)
  - Otherwise → advance one step
- `restoreNav()` re-creates the `tut-tap-hint` div if `navEl.innerHTML` was replaced with the play button on the last page. Called at the top of every `render()`.
- `setNavActive(active)` toggles the `.visible` class on the hint element.

**CSS for the hint:**

```css
.tut-tap-hint {
    font-family: 'Space Mono', monospace;
    font-size: 1.8cqw;
    letter-spacing: 0.12em;
    color: #071b1e;
    opacity: 0;
    transition: opacity 0.4s ease;
    animation: tap-pulse 2.4s ease-in-out infinite;
    text-transform: lowercase;
    user-select: none;
}
.tut-tap-hint.visible { opacity: 0.4; }
@keyframes tap-pulse {
    0%, 100% { opacity: 0.4; }
    50%       { opacity: 0.2; }
}
```

**`tutorial.html` bug fixed:** the click listener was attached via `getElementById('tut-container')` but that div has a class, not an id — returned `null` so the handler was never registered. Fixed with `querySelector('.tut-container')`.

### 5. "Click to Continue" Timing Fixes

Two bugs caused the prompt to appear before animations were done:

**Bug A — render paths calling `setNavActive(true)` unconditionally:** The `keepTiles` and simple-fade render paths called `setNavActive(true)` right after `await applyStep()` returned. But `applyStep` may have synchronously set `introPlaying = true` (e.g. page 2's tile animation). Fixed by guarding all render-path calls with `if (!introPlaying) setNavActive(true)`.

**Bug B — page 3 flower flips had no `introPlaying` lock:** The 5 flower cards use CSS `animationDelay` (`0.35 + flipIdx * 0.6s`, last flip completes at ~3.75s) but `introPlaying` was never set, so the prompt appeared immediately. Fixed in `applyStep`'s `if (rows.length > 0)` block:

```javascript
introPlaying = true;
setNavActive(false);
var lastFlowerMs = 350 + (flips.length - 1) * 600 + 1000 + 150; // 3900ms
setTimeout(() => {
    introPlaying = false;
    setNavActive(true);
}, lastFlowerMs);
```

The `else` branch (already-HTML rows, navigating backward) correctly skips this lock so the prompt appears immediately on revisit.

---

## Key Invariants

- `introPlaying = true` must always be paired with a corresponding `introPlaying = false` + `setNavActive(true)` in a timeout callback.
- `setNavActive(true)` in render transition paths must be guarded with `if (!introPlaying)` — `applyStep` may have already started a locked animation.
- `restoreNav()` must run at the top of every `render()` call to handle the last-page play button replacement.
- Both `index.html` and `tutorial.html` contain full duplicate copies of all tutorial CSS and JS — every change must be applied to both files.
